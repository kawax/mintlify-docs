## What is Workbench?

Orchestra Testbench Workbench creates a complete Laravel application inside your package repository. Separate from your automated tests (PHPUnit/Pest), it lets you verify package behavior interactively through a browser or HTTP client.

```mermaid
flowchart TD
    A["Package code<br>`src/`"] --> B["Workbench<br>`workbench/`"]
    B --> C["WorkbenchServiceProvider<br>demo service registration"]
    B --> D["Routes and controllers<br>browser verification"]
    B --> E["Migrations and seeders<br>realistic data testing"]
    B --> F["`composer serve`<br>local server runtime"]
    C --> G["testbench.yaml<br>providers / build config"]
```

Workbench lets you:

- Run your package inside a real Laravel application context
- Create routes and views to manually verify UI and API behavior
- Populate realistic test data using seeders for integration testing
- Start a local development server with `composer serve`

## Installation and setup

<Steps>
  <Step title="Install Orchestra Testbench">
    ```bash
    composer require --dev orchestra/testbench
    ```
  </Step>
  <Step title="Run the workbench:install command">
    ```bash
    vendor/bin/testbench workbench:install
    ```

    This command does the following in one step:
    - Creates the `workbench/` directory structure
    - Adds the Workbench namespaces to `composer.json` `autoload-dev`
    - Adds build scripts to `composer.json` `scripts`

    <Tip>
    Additional options:
    - `--force`: overwrite existing files
    - `--basic`: minimal setup without routes and package discovery configuration
    - `--devtool`: enable DevTool support
    </Tip>
  </Step>
  <Step title="Review the composer.json additions">
    After installation, `composer.json` includes the following:

    ```json
    {
        "autoload-dev": {
            "psr-4": {
                "Tests\\": "tests/",
                "Workbench\\App\\": "workbench/app/",
                "Workbench\\Database\\Factories\\": "workbench/database/factories/",
                "Workbench\\Database\\Seeders\\": "workbench/database/seeders/"
            }
        },
        "scripts": {
            "post-autoload-dump": [
                "@clear",
                "@prepare"
            ],
            "clear": "@php vendor/bin/testbench package:purge-skeleton --ansi",
            "prepare": "@php vendor/bin/testbench package:discover --ansi",
            "build": "@php vendor/bin/testbench workbench:build --ansi",
            "serve": [
                "Composer\\Config::disableProcessTimeout",
                "@build",
                "@php vendor/bin/testbench serve --ansi"
            ]
        }
    }
    ```
  </Step>
  <Step title="Initialize the Workbench environment">
    ```bash
    composer clear
    composer prepare
    composer build
    ```
  </Step>
</Steps>

## Configuring testbench.yaml

`testbench.yaml` is the main configuration file that controls Workbench behavior. Create it in your package root.

<Warning>
`testbench.yaml` often contains environment-specific settings. Add it to `.gitignore` and keep a `testbench.yaml.example` template in the repository for other developers.
</Warning>

### Basic configuration

```yaml
providers:
  - Vendor\Package\PackageServiceProvider
  - Workbench\App\Providers\WorkbenchServiceProvider

migrations:
  - workbench/database/migrations

workbench:
  start: '/'
  install: true
  health: false
  discovers:
    web: true
    api: false
    commands: false
    components: false
    views: false
    config: true
  build:
    - asset-publish
    - create-sqlite-db
    - db-wipe
    - migrate-fresh:
        --seed: true
        --seeder: Workbench\Database\Seeders\DatabaseSeeder
  assets:
    - laravel-assets
```

### Configuration reference

| Key | Description |
|---|---|
| `providers` | Service providers to load in the Workbench environment |
| `migrations` | Migration directories to run |
| `workbench.start` | Default URL when running `composer serve` |
| `workbench.install` | Whether to run package installation during build |
| `workbench.health` | Enable health checks |
| `workbench.discovers` | Control Laravel package auto-discovery |
| `workbench.build` | Commands to run during the build step |
| `workbench.assets` | Assets to publish |

### Environment variables

```yaml
env:
  - APP_ENV=testing
  - APP_KEY=base64:your-app-key
  - DB_CONNECTION=sqlite
  - DB_DATABASE=:memory:
  - YOUR_PACKAGE_CONFIG=value
```

<Tip>
Commit a `testbench.yaml.example` to your repository so other developers can quickly set up their Workbench environment.
</Tip>

## Creating Workbench components

### WorkbenchServiceProvider

Create a service provider to register Workbench-specific services and wire up demo routes and views.

```php
<?php

namespace Workbench\App\Providers;

use Illuminate\Support\ServiceProvider;
use Vendor\Package\SomeClass;
use Workbench\App\Services\DemoService;

class WorkbenchServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        $this->app->singleton(DemoService::class);
    }

    public function boot(): void
    {
        // Register package extensions or demo functionality
        SomeClass::register('demo', DemoService::class);

        // Load Workbench routes
        $this->loadRoutesFrom(__DIR__.'/../../routes/web.php');

        // Load Workbench views
        $this->loadViewsFrom(__DIR__.'/../../resources/views', 'workbench');
    }
}
```

### DatabaseSeeder

Create a seeder to populate demo data.

```php
<?php

namespace Workbench\Database\Seeders;

use Illuminate\Database\Seeder;
use Workbench\Database\Factories\UserFactory;

class DatabaseSeeder extends Seeder
{
    public function run(): void
    {
        UserFactory::new()->count(10)->create();

        $this->call([
            PackageDataSeeder::class,
        ]);
    }
}
```

### Model factories

Create factories to generate test and demo data.

```php
<?php

namespace Workbench\Database\Factories;

use Illuminate\Database\Eloquent\Factories\Factory;
use Workbench\App\Models\User;

class UserFactory extends Factory
{
    protected $model = User::class;

    public function definition(): array
    {
        return [
            'name' => $this->faker->name(),
            'email' => $this->faker->unique()->safeEmail(),
            'email_verified_at' => now(),
            'password' => bcrypt('password'),
        ];
    }
}
```

### Controllers and routes

Create controllers to demonstrate package functionality.

```php
<?php

namespace Workbench\App\Http\Controllers;

use Illuminate\Http\Request;
use Vendor\Package\Facades\YourPackage;

class DemoController extends Controller
{
    public function index()
    {
        $data = YourPackage::getData();

        return view('workbench::demo', compact('data'));
    }

    public function api()
    {
        return response()->json([
            'package_version' => YourPackage::version(),
            'features' => YourPackage::getFeatures(),
        ]);
    }
}
```

Define routes in `workbench/routes/web.php`:

```php
<?php

use Illuminate\Support\Facades\Route;
use Workbench\App\Http\Controllers\DemoController;

Route::get('/', [DemoController::class, 'index']);
Route::get('/api/demo', [DemoController::class, 'api']);
```

### Console commands

Define Artisan commands in `workbench/routes/console.php`:

```php
<?php

use Illuminate\Foundation\Inspiring;
use Illuminate\Support\Facades\Artisan;

Artisan::command('inspire', function () {
    $this->comment(Inspiring::quote());
})->purpose('Display an inspiring quote');
```

Run Workbench console commands through `vendor/bin/testbench`:

```bash
vendor/bin/testbench inspire
vendor/bin/testbench list
```

## Testing with the WithWorkbench trait

Use the `WithWorkbench` trait to apply `testbench.yaml` configuration to your automated test suite.

```php
<?php

namespace Tests;

use Orchestra\Testbench\Concerns\WithWorkbench;
use Orchestra\Testbench\TestCase as Orchestra;

abstract class TestCase extends Orchestra
{
    use WithWorkbench;

    protected function getPackageProviders($app): array
    {
        return [
            \Vendor\Package\Providers\YourServiceProvider::class,
        ];
    }

    protected function defineEnvironment($app): void
    {
        $app['config']->set('database.default', 'testing');
        $app['config']->set('your-package.key', 'test-value');
    }
}
```

### Feature test example

```php
<?php

namespace Tests\Feature;

use Tests\TestCase;

class WorkbenchIntegrationTest extends TestCase
{
    public function test_workbench_routes_are_accessible(): void
    {
        $response = $this->get('/');

        $response->assertStatus(200);
        $response->assertViewIs('workbench::demo');
    }

    public function test_package_api_integration(): void
    {
        $response = $this->get('/api/demo');

        $response->assertStatus(200);
        $response->assertJsonStructure([
            'package_version',
            'features',
        ]);
    }
}
```

<Info>
Using `WithWorkbench` shares configuration between your automated tests and the interactive Workbench app, keeping both environments consistent.
</Info>

## Development workflow

### Daily development

```bash
# Start the development server
composer serve

# Rebuild after changes
composer build

# Run tests
composer test
```

### How `composer serve` works

`composer serve` internally runs:

1. `composer build` to rebuild the Workbench application
2. `vendor/bin/testbench serve` to start the local server

Access your Workbench app at `http://localhost:8000`.

### The package development cycle

<Steps>
  <Step title="Write package code">
    Implement your package logic in `src/`.
  </Step>
  <Step title="Update testbench.yaml">
    Configure the required providers and build steps.
  </Step>
  <Step title="Build Workbench">
    ```bash
    composer build
    ```
  </Step>
  <Step title="Verify behavior">
    Use Workbench routes and controllers to check behavior in a browser or HTTP client.
  </Step>
  <Step title="Iterate">
    Run `composer build` after each change to update the Workbench app.
  </Step>
</Steps>

## Advanced configuration

### Asset management

For packages with frontend assets:

```yaml
workbench:
  assets:
    - laravel-assets
    - package-assets
  build:
    - asset-publish
    - npm-install
    - npm-run-build
```

### Custom database configuration

To use a database other than SQLite in-memory:

```yaml
env:
  - DB_CONNECTION=mysql
  - DB_HOST=127.0.0.1
  - DB_PORT=3306
  - DB_DATABASE=workbench_test
  - DB_USERNAME=root
  - DB_PASSWORD=

migrations:
  - workbench/database/migrations
  - vendor/other-package/migrations
```

## Troubleshooting

### Debug commands

```bash
# Clear and do a full rebuild
composer clear && composer prepare && composer build

# Check package discovery
vendor/bin/testbench package:discover --ansi

# Inspect configuration
vendor/bin/testbench about

# List all routes
vendor/bin/testbench route:list
```

### Common issues

<AccordionGroup>
  <Accordion title="Workbench fails to build">
    Check `testbench.yaml` syntax and provider configuration. YAML indentation errors are a common cause.
  </Accordion>
  <Accordion title="Routes are not loading">
    Verify the route file paths and the WorkbenchServiceProvider registration. Confirm that `discovers.web` is set to `true` in `testbench.yaml`.
  </Accordion>
  <Accordion title="Database errors">
    Check that migration paths are correct and the database is properly configured. When using SQLite, make sure the `create-sqlite-db` build step is included.
  </Accordion>
  <Accordion title="Assets are not published">
    Check the `workbench.assets` and the `asset-publish` entry in `workbench.build`.
  </Accordion>
</AccordionGroup>

## Best practices

- **Keep Workbench code separate from package code**: `workbench/` is for demo and verification only; keep package logic in `src/`
- **Manage environment variables in `testbench.yaml` `env:` rather than `.env`**
- **Commit `testbench.yaml.example`** so other developers can set up quickly
- **Use Workbench and automated tests for different purposes**: automate regression tests in `tests/`, use `workbench/` for end-to-end and interactive verification
- **Leverage composer scripts**: use `composer serve` and `composer build` to streamline your development workflow
